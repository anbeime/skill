## 概览
本文档定义了多智能体会议决策Skill的标准输出格式。会议记录采用Markdown格式，清晰展示会议基本信息、完整发言过程、决策结论和后续行动。

## 会议记录结构

```
1. 会议基本信息
2. 参会智能体
3. 会议发言过程
   - 阶段一：开场发言
   - 阶段二：自由讨论
   - 阶段三：深入辩论
   - 阶段四：共识收敛
   - 阶段五：决策生成
4. 决策结论
5. 关键论据汇总
6. 风险提示
7. 后续行动建议
```

## 各部分详细说明

### 1. 会议基本信息
- **会议主题**: 简洁描述会议讨论的核心问题
- **会议时间**: 模拟的会议时间（可选）
- **会议目标**: 期望达成的决策类型
- **关键约束**: 需要考虑的限制条件

### 2. 参会智能体
以表格或列表形式展示所有参会智能体的角色信息：
- 角色名称
- 专业领域
- 性格特点

### 3. 会议发言过程
按照时间顺序记录各阶段的发言内容，每个发言包含：
- **发言者**: 智能体角色名称
- **发言内容**: 完整的发言内容
- **发言要点**（可选）: 提炼的关键点

**格式规范**:
- 使用引用块 `>` 标记每个发言
- 清晰标注发言阶段和序号
- 保持发言的原貌和逻辑

### 4. 决策结论
简洁明了地陈述最终达成的决策结论：
- 明确的决策结果（是/否/选择哪个方案）
- 决策的主要依据
- 决策的适用范围和条件

### 5. 关键论据汇总
汇总支持决策的主要论据，按照观点类型分类：
- **支持方论据**: 3-5个核心论据
- **反对方论据**: 被否定的主要论据
- **折中方案**: 最终采用的折中思路

### 6. 风险提示
识别决策存在的潜在风险：
- **技术风险**: 技术实现方面的风险
- **商业风险**: 市场和商业方面的风险
- **运营风险**: 执行和运营方面的风险
- **合规风险**: 法律和合规方面的风险
- **缓解措施**: 针对每个风险的建议应对方案

### 7. 后续行动建议
提供具体的后续行动建议：
- **立即行动**: 需要立即执行的步骤
- **短期计划**: 未来1-3个月的行动计划
- **长期规划**: 中长期的发展方向
- **责任分工**: 建议的分工方式（如适用）

## 格式示例

```markdown
# 会议记录：微服务架构可行性评估

## 1. 会议基本信息
- **会议主题**: 是否将现有单体架构改造为微服务架构
- **会议目标**: 评估微服务架构的可行性并形成决策建议
- **关键约束**: 预算500万，时间6个月，团队规模20人

## 2. 参会智能体
| 角色 | 专业领域 | 性格特点 |
|------|---------|---------|
| 技术架构师 | 系统架构、技术选型 | 理性严谨、追求卓越 |
| DevOps工程师 | 部署运维、CI/CD | 实用主义、风险意识强 |
| 前端工程师 | 用户体验、前端性能 | 用户导向、注重体验 |
| CTO | 技术战略、资源决策 | 战略思维、平衡风险 |

## 3. 会议发言过程

### 阶段一：开场发言

> **主持人**: 今天我们讨论是否采用微服务架构。请各位从各自专业角度发表意见。

> **技术架构师**: 从架构角度看，当前单体系统已达到500万行代码，部署周期长达2小时。微服务可以带来更好的可扩展性和团队自治，但需要处理服务间通信和数据一致性问题。我倾向于谨慎推进。

> **DevOps工程师**: 运维角度，微服务会增加部署复杂度和监控成本。目前我们的自动化测试覆盖率只有60%，如果拆分成50+服务，管理成本会成倍增加。建议先提升基础设施能力。

> **前端工程师**: 用户体验方面，我们期望实现秒级响应和独立功能迭代。微服务可以让前端团队更灵活，但API接口碎片化会增加对接复杂度。建议先试点核心服务。

> **CTO**: 综合来看，技术方向是对的，但需要评估投入产出比。500万预算能否支撑6个月完成改造？20人团队的技能储备是否足够？

### 阶段二：自由讨论
（省略详细发言内容，保留关键观点）

### 阶段三：深入辩论
（省略详细发言内容，保留关键观点）

### 阶段四：共识收敛
（省略详细发言内容，保留关键观点）

### 阶段五：决策生成

> **主持人**: 综合各位的讨论，我们达成以下共识...

## 4. 决策结论
**采用渐进式微服务改造方案**，具体决策如下：

1. **第一阶段（3个月）**: 试点拆分用户服务和订单服务，验证技术可行性
2. **第二阶段（6个月）**: 拆分5-8个核心服务，建立微服务基础设施
3. **暂缓全面改造**: 在验证成功前，暂停其他服务的拆分

**决策依据**:
- 技术方向正确，但当前团队和基础设施储备不足
- 渐进式方案可以控制风险，验证成功后再扩大
- 符合预算和时间约束，避免一次性投入过大

## 5. 关键论据汇总

### 支持方论据
1. 单体系统已达500万行代码，维护困难，部署周期长
2. 微服务可以提高团队开发自治性和迭代速度
3. 技术趋势和行业标准支持微服务架构

### 反对方论据
1. 当前自动化测试覆盖率不足，微服务会增加管理成本
2. 团队技能储备不足，需要培训成本
3. 数据一致性和服务间通信是技术难点

### 折中方案
采用渐进式改造，先试点验证，避免一次性投入过大风险

## 6. 风险提示

| 风险类型 | 风险描述 | 风险等级 | 缓解措施 |
|---------|---------|---------|---------|
| 技术风险 | 服务间通信延迟影响性能 | 中 | 引入服务网格和缓存机制 |
| 运营风险 | 团队技能不足影响进度 | 高 | 提前培训，引入外部专家 |
| 成本风险 | 基础设施投入超出预算 | 中 | 采用云服务按需付费 |

## 7. 后续行动建议

### 立即行动（本周内）
- [ ] 成立微服务改造专项小组
- [ ] 评估现有系统，制定拆分方案
- [ ] 启动团队技术培训计划

### 短期计划（3个月内）
- [ ] 完成用户服务微服务化改造
- [ ] 建立CI/CD基础设施
- [ ] 完成服务监控和日志系统

### 长期规划（6-12个月）
- [ ] 拆分5-8个核心服务
- [ ] 建立微服务治理体系
- [ ] 评估是否全面推广
```

## 输出规范

### 格式要求
- 使用Markdown格式输出
- 标题层级清晰，不超过4级
- 使用表格展示结构化信息
- 引用块用于发言内容

### 内容质量
- 发言内容真实自然，符合角色设定
- 讨论过程有逻辑，观点交锋明显
- 决策结论明确，论据充分
- 风险识别全面，措施具体可行
- 行动建议可落地，时间节点清晰

### 长度控制
- 完整会议记录建议不超过3000行
- 发言过程部分可根据需要展开或精简
- 决策结论和关键论据应简明扼要
- 风险和行动建议应具体但不冗余
